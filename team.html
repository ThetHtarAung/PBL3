2. Introduction

*The Bicycle Parking Lot Occupancy Locator Web Application is designed to provide users with real-time occupancy updates and historical data on bicycle parking lot availability and other relevant functions. This document described the functional and non-functional requirements for the implementation of this project.

2.1. Purpose

*The purpose of this system is to improve the user experience and facilitate effective use of bicycle parking facilities through real time updates on occupancy rates in selected parking spaces. The application seeks to shorten the time it takes users to find available parking spaces and encourages sustainable transport choice by providing them with timely information.

2.2. Scope

*The application will provide users with the information they need on bicycle parking facilities, including their locations, capacities, costs and if necessary RealtimeOccupancy Rate. In addition, users will be able to choose parking spaces on the basis of their preferences, e.g. distance, occupancy conditions and user inclinations towards Free Parking Areas. In addition, the system will help users to find their chosen parking spaces more effectively by providing navigational assistance.

4. User Requirements

4.1. Functional Requirements

4.1.1. User Registration

*The system shall allow account registration and log in for each user.

*The system shall store user registration information.

4.1.2. Real-Time Occupancy Updates

*The system shall display levels of occupancy.

*e.g.. Crowded, Neutral, Not Crowded.

*The data of occupancy should be based on the historical record dataset.

4.1.3. Parking Lot Profiles

*Each parking lot shall have its location, capacity, price, security of the facility and level of occupancy.

*Users should be able to search and filter parking lots based on specific criteria.

4.1.4. Historical Data Analysis

*The system shall analyze historical occupancy data and predict occupancy based on the data of time taken.

4.1.5. User Feedback and Ratings

*Users shall be able to provide feedback and ratings for bicycle parking lots.

*Based on the feedback, the system shall suggest the best parking lots around the users.

4.2. Non-functional Requirements

4.2.1. Performance

*The system should be responsive and capable of handling simultaneous user requests.

*Real-time updates based on the data analysis should be processed efficiently to provide the latest information.

4.2.2. Speed

*The application shall not exceed ten seconds to respond to the user request.

4.2.3. Scalability

*The system architecture should be scalable to accommodate future growth and expansion.

**Infrastructure and resources should be flexible to handle increased user traffic and data volume.

4.2.4. Usability

*The user interface should be friendly to all users from youths to elderly.


5. System Requirements

5.1. Functional Requirements

5.1.1. User Registration

*The system shall store user registration information.

*Registration process must include security measures such as Email verification.

5.1.2. User Login

*Users shall be accessed their individual profiles when the correct Credentials are inputted into the login screen.

*The system shall grant permission to the user to re-assign their password once the correct email of the registered account is provided.

5.1.3. Occupancy Prediction

*The system shall be able to determine the level of occupancy.

*The system shall update occupancy status to the latest information.

5.1.4. Route Navigation

*The system shall guide its users to the correct path in order to get to the selected parking lot.

*Alternative routes shall be suggested taking in consideration traffic updates.

5.1.5. Parking Lot Profiles

*Display basic information about parking lots such as its name, location, and rates shall be displayed by the system.

*The system shall make updates on any relevant changes made in the parking lot profile.

5.1.6. User Feedback and Ratings

*The system shall store user feedback into a database.

*The system shall update a parking lot’s rating according to the user's ratings report.

5.2. Non-functional Requirements

5.2.1 Product Requirements

5.2.1.1. Performance

*The system shall respond swiftly when users requests from the system (e.g navigation, registration) under normal system conditions.

*The system shall have a consistent performance when dealing with a large number of users active at the same time.

*The system shall provide an error message and reasoning if or when the user encounters an error the system cannot resolve.

5.2.1.2. Security

*Access to administrative controls shall be prohibited from non-administrative users (e.g changing application content, tempering with code, etc).

*Sensitive information such as users password, email and location shall be encrypted.

5.2.1.3. Usability

*User interface must be easy to understand by users of varying technical skills.

5.2.2. Organizational Requirements

5.2.2.1. Collaboration

*The system shall facilitate internal collaboration between department teams such as IT, customer service and operations.

*The feedback database should be able to be accessed by administrative personnel.

5.2.3. External Requirements

5.2.3.1. Regulatory Compliance

*The system shall comply with relevant industry standards such as data protection laws and parking laws

5.2.3.2. Localization

*The system shall support multi-language settings.

*The system shall allow customization by governments to meet requirement of different regional laws.


6. System Architecture

6.1. Client-Side Application

*It has the user interface (UI) for interacting with the web application that runs in the user’s web browser.

*It allows users to search for parking lots profiles, view real-time occupancy updates, analyze historical data, and provide feedback.

*The users send requests to the server-side application to retrieve data and perform various actions such as route navigation or submitting feedback.

6.2. Server-Side Application

*The server-side application is to handle incoming requests from the client-side application, processing data, and sending responses back to the client.

*It consists of the following components:

*Web Server: Handles the incoming HTTP requests and routes them to the appropriate destination.

*Database: Stores data, such as user accounts, parking lot information, historical occupancy data, and user feedback.

*External APIs: SInterfaces with mapping APIs (Google Maps) for route navigation and location-based parking lot profiles and occupancy.

*The server-side application interacts with the database to store and retrieve data, perform data analysis to predict occupancy rate and with APIs to show the required information on map.

8.1 Application Requirements

8.1.1. Configuration

*Operating System: Linux/Windows/MacOS

*Web Server: Apache/Nginx

*Programming Language: Python 3.7+

*Framework: Django 3.0+

*Browser Compatibility: Latest versions of Chrome, Firefox, Safari, Edge.

8.2 Database Requirements

8.2.1. Tables

*Parking Locations: It contains details of all bicycle parking spots, including unique IDs, addresses, and GPS coordinates.

*Occupancy Status: It records the current occupancy status of each parking location, with timestamps.

*User Information: Stores user details such as user’s bicycle number and ID information.

8.2.2. Relationships:

*Each record in the Occupancy_Status table is linked to a specific parking spot in the Parking_Locations table via foreign keys (Station_ID) and (Timestamp).

*The User_Info table is related to the Occupancy_Status table through a foreign key (Bicycle_ID) to log and manage user interactions.
