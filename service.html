Introduction

This document is written as System Design Diagram Document for the “Bicycle Parking Lots Occupancy Locator Web Application”. It aims to illustrate how functional and non-functional which are documented in the Requirement Specifications Document can be transformed into a working application.


System Architecture

The system architecture for the parking lot management system is organized into three main tiers: the user interface (presentation tier), backend services (application tier), and database (data tier). Additionally, the architecture includes external APIs such as Google Maps API and Route Calculator API.

UML Activity Diagram

Selecting parking lot profile

Description

1. After entering the application, the interface will display a recommended parking lot list based on relative distance between the user and parking lot.
2. The system will retrieve users’ current location and query database for nearby parking lots and sorts parking lots by distance between the user.
3. Then the recommended parking lots will be displayed with brief information including names, distances, and available spaces.
4. Users will have to decide whether to choose from the displayed list or not.
4.1 If the user chooses the recommended list, the information will display. If not, they'll use the search function to enter criteria and filter parking lots accordingly.
5. Users will choose a parking lot from the displayed list.
6. After choosing, the chosen parking lot ID will be stored and the parking lot information will be displayed to users while the system retrieves detailed information of the selected parking lot.
7. Users will have to decide whether to use this parking lot or not.
7.1 If chosen, proceed with route navigation; if not, select another parking lot.

Route Navigation

Description

1. The route navigation system will calculate routes initially.
2. The system will retrieve the user's current location, and call Google Maps API with current and destination locations. Then the system will receive and parse the route data.
3. A set of turn-by-turn route instructions will be generated and be displayed to users
4. Users will be asked to follow the route instructions. Meanwhile, the system will monitor the user's location continuously to check whether the user is on the current route or not.
4.1 If on the correct route, continue navigation; if not, trigger route recalculation for new turn-by-turn instructions.
5. The turn-by-turn instructions will be continuously displayed to the user.
6. When the system detects the user is close to the destination, the user will choose whether he/she arrives at the destination or not.
6.1 If yes, user has arrived and navigation ends; if not, trigger route recalculation.

Occupancy Rate

Description

1. After the user selects a parking lot to be navigated, the system will confirm the current time with the time in the historical data.
2. The system will calculate the average occupancy rate based on the historical data by dividing predicted occupied lots by total capacity.
3. Check if the occupancy rate is below 25% by multiplying the result by 100.
3.1 If yes, then the system will display “Uncrowded” in green font to tell users that the selected parking lot is most likely to be available
4. If no, then the system will re-calculate if the occupancy rate is below 50%
4.1 If yes, then the system will display “Less Crowded” in yellow font to tell users that the selected parking lot is likely to be available.
5. If no, then the system will re-calculate if the occupancy rate is below 75%
5.1 If yes, then the system will display “Relatively Crowded” in orange font to tell the user that the selected parking lot has a risk of being fully occupied.
6. If no, then it suggests that the predicted occupancy rate is in between 75% to 100%. Therefore, the system will display “Crowded” in red font to tell users that the selected parking lot has a high risk of being fully occupied.
7. After the display of the predicted occupancy rate, the user decides if he/she is still willing for the navigation to the selected parking lot.
7.1 If yes, then start navigation to the selected parking lot, If no, then go back to the first step and select a new parking lot.

Description

Register/Login: Users register new accounts and log into the system to access personalized features and data.

Manage parking lots information: Manage and maintain information related to parking lots to ensure data accuracy and completeness.
Manage occupancy data: Monitor and manage parking lot occupancy data to ensure real-time and accurate data.

Monitor system performance: Monitor the system's operational status to ensure its stability and reliability and address anomalies promptly.

Ensure data clarity: Ensure that all relevant data is collected and uploaded to provide parking lot information.

2. Sequence Diagram

2.1 Parking Lot Profile Sequence Diagram

Description

1. After entering into the application, the user will be shown different parking lots with information and profiles of their infrastructure. 2. When a user wants to access a certain parking lot, the user interface will request the system to access the lot profile 3. The system will query the lot profile from the database and display it to the UI via the profile system database. The user will retrieve the data from there. In case of updated data: 4. [if data need update]: If the system updates the parking lot profile data, send it into the database, the system then would send the data to the UI 5. The latest updates are shown to the users for access.

2.2 Occupancy Rate Sequence Diagram

Description

1. When a user wants to view the occupancy rate, the user views info with ViewInfo, then the parking UI sends a request to the system and queries the occupancy rate from the database.
2. The database then returns occupancy rate and sends the rate data to the UI.
Alternative block:
3. [if data need update]: If the system updates the occupancy data, send it into the database, the system then would send the data to the UI.
4. At last, the UI would display the occupancy rate to the user.

2.3 Route Navigation Sequence Diagram

Description

1. After entering into the application, the user starts by selecting the start and destination points. The web application collects this user input.Following this, the web application sends a request to the Google Map API to get basic map data.
2. The Google Maps API responds by returning the map data and basic directions to the web application.Next, the web application sends a request to the Route Calculator API for route calculation and recommendations.
3. The Route Calculator API processes this request and returns the calculated route along with alternative routes to the web application. The web application then updates the Google Map with the calculated route. Finally, the user can view the updated map with the recommended route and alternative routes.
In case of updated data: 3. [if data need update]: If the system updates the route data, the system then would send the data to the UI. 4. At last, update it to the UI for users to access the updated and new ones.

Behavioral Model

1. Activity Model

1.1 Selecting parking lot profile

Description

In this activity model, users first authorize access to their location information. The system uses this location information to identify nearby parking lots. After obtaining the data, the system extracts the occupancy rate and profile information of these parking lots. This information is presented to the user, who then decides whether to continue the navigation.

1.2 Route navigation

Description

In this activity model, users provide their location information. The system uses this information to access the data of the selected parking lot. The system then calls the Google Maps API to calculate the route from the user's current location to the selected parking lot. The user navigates according to the navigation instructions provided by the system, and the system confirms whether the user has reached the destination.

1.3 Occupancy Rate

Description

The activity model of occupancy starts with access to a historical database to retrieve parking lot profile information and historical data. The system processes the parking lot data, calculates the current occupancy rate, and provides the current occupancy rate of the bicycle parking lot. The system uses the calculated occupancy rate to determine the occupancy level. Based on the level of occupancy, the user decides which parking lot to navigate to and starts navigation accordingly.
